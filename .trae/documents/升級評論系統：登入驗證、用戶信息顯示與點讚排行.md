為了實現評論區的升級，我們需要進行以下步驟：

### 1. 資料庫模型升級 (`prisma/schema.prisma`)
- **新增 `likes` 欄位**：在 `AlbumReview` 模型中新增 `likes` (Int) 欄位，記錄點讚數。
- **新增 `ReviewLike` 模型**（可選但推薦）：為了防止重複點讚，最好新增一個關聯表 `ReviewLike` (`userId`, `reviewId`)。但為了簡化，本次先只在 `AlbumReview` 增加 `likes` 計數，並在前端做簡單的 localStorage 防抖，或者後端簡單處理。**決定方案：** 為了體驗更好，我們還是做一個簡單的 `ReviewLike` 關聯表，確保每人對每條評論只能讚一次。
- **執行更新**：`npx prisma db push` 和 `npx prisma generate`。

### 2. 後端 API 升級
- **評論列表 API (`app/api/reviews/route.ts`)**：
    - **獲取評論 (GET)**：
        - 需要從 Clerk 獲取每個評論的 `userId` 對應的頭像 (`imageUrl`) 和暱稱 (`firstName` / `username`)。這需要在後端批量查詢 Clerk API。
        - 根據點讚數 (`likes` 數量) 進行降序排序。
        - 返回數據需包含：評論內容、評分、點讚數、**是否已點讚**（針對當前登入用戶）、用戶頭像、暱稱。
    - **發表評論 (POST)**：
        - 強制檢查 `userId`，確保只有登入用戶能發送（現有邏輯已包含 `getUserIdOr401`）。
- **點讚 API (`app/api/reviews/[id]/like/route.ts`)**：
    - 新增一個路由處理點讚/取消點讚。
    - 邏輯：檢查 `ReviewLike` 表，若存在則刪除（取消讚）並減 1，若不存在則創建並加 1。

### 3. 前端組件升級 (`components/AlbumGrid.tsx`)
- **顯示優化**：
    - 評論列表項中加入用戶頭像（圓形）和暱稱。
    - 加入點讚按鈕（愛心圖標），顯示當前點讚數。
- **排序邏輯**：前端收到數據後，默認已經是按點讚數排序的，但也許需要處理「點讚後即時重排」或「點讚後數字更新」的 UI 邏輯。
- **登入狀態檢查**：
    - 使用 Clerk 的 `useUser` 或 `useAuth`。
    - 如果未登入，評論輸入框顯示「請先登入後評論」，並禁用發送按鈕或跳轉登入。

**重點提示**：
由於資料庫不存用戶信息，後端 `GET /api/reviews` 將會是性能瓶頸（需要多次請求 Clerk）。為優化，我們可以使用 `clerkClient.users.getUserList` 批量獲取 ID 列表對應的用戶信息。
